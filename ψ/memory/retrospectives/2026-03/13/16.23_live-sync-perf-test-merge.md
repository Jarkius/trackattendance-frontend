### Key Code Changes
- `sync.py`: Removed `self.db.mark_scans_as_synced([scan.id])` from `sync_single_scan()` — this method runs in a daemon thread but the SQLite connection belongs to the main thread. Batch sync marks it later; idempotency keys prevent double-counting.
- `tests/test_live_sync.py`: Tests cover dup found/not-found, fail-open on HTTP/timeout/connection errors, read-only skip, custom params, successful sync, explicit no-SQLite-touch assertion, error handling.
- `tests/perf_live_sync.py`: 4 tests — single latency (10 sequential), 20 concurrent dup checks, 20 concurrent syncs, paced mixed workload. Handles rate limiting gracefully, separates 429s from real errors.

### Architecture Decisions
- **Don't mark synced from background thread**: Let batch sync handle marking. Idempotency keys ensure no double-counting. Simpler than creating a new SQLite connection per thread or using `check_same_thread=False`.
- **Performance thresholds**: Sequential <200ms (real per-station), concurrent <600ms (single-IP contention artifact). In production each station has its own IP.

## Lessons Learned
- **Pattern**: When testing from single IP against rate-limited APIs, separate rate-limit errors from real errors — they're different failure modes
- **Discovery**: SQLite threading constraint is best handled by removing the cross-thread DB call entirely rather than working around it — simpler, fewer failure modes
- **Pattern**: Performance test thresholds must account for test infrastructure limitations (single IP, shared connection pool) vs production reality (distributed stations, separate IPs)
